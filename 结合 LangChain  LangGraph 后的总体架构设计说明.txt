# 总体架构设计（结合 LangChain / LangGraph）

## 1. 设计目标

本系统的目标，不是只做一个简单的聊天机器人，也不是只做一个通用 RAG 问答系统，而是构建一套面向本地生活 / 商品推荐 / 活动推荐场景的智能检索与推荐系统。

系统需要同时支持以下几类能力：

1. 商品、活动、门店的结构化搜索与筛选
2. 基于自然语言的推荐与导购
3. 商品规则、活动说明、FAQ 等知识问答
4. 库存、价格、营业状态等实时确认
5. 多工具协同下的 Agent 式交互

结合 LangChain / LangGraph 后，整体架构应当采用：

**“底层能力层 + LangChain / LangGraph Agent 编排层”** 的分层模式。

其中：

* 底层能力层负责提供稳定、可控、可查询的数据与服务能力
* LangChain / LangGraph 负责意图理解、参数抽取、工具选择、结果整合和回答生成

LangChain 官方当前主要提供 agents、tools、structured output、retrieval 等能力，用于构建 LLM 应用和工具编排流程；LangGraph 则更适合承载工作流、状态管理和更复杂的 agent 执行链路。

---

## 2. 总体分层

系统总体上分为四层：

1. 业务数据源层
2. 底层能力层
3. LangChain / LangGraph Agent 编排层
4. 用户交互层

可用下图表示：

```text
用户交互层
    -> App / 小程序 / Web / Chat UI

LangChain / LangGraph Agent 编排层
    -> Query Parser
    -> Tool Router
    -> Response Composer

底层能力层
    -> 结构化层
    -> 知识层
    -> 实时层

业务数据源层
    -> 商品中心
    -> 活动中心
    -> 门店中心
    -> 类目系统
    -> 标签系统
    -> 价格系统
    -> 库存系统
    -> 规则文档 / FAQ / 攻略内容
```

---

## 3. 业务数据源层

业务数据源层负责提供系统运行所需的原始业务真相数据。
这一层包括但不限于：

* 商品中心
* 活动中心
* 门店中心
* 类目系统
* 标签系统
* 价格系统
* 库存系统
* 审核系统
* 规则文档 / FAQ / 攻略内容

这一层的特点是：

* 数据真实
* 表结构复杂
* 写入频繁
* 面向交易、审核、运营
* 不适合直接给 Agent 或搜索服务高频访问

因此，业务数据源层不直接暴露给 LangChain Agent，而是先经过底层能力层整理和封装。

---

## 4. 底层能力层

底层能力层是整个系统的核心基础设施，负责把业务真相加工成可供 Agent 调用的稳定能力。

底层能力层分为三部分：

### 4.1 结构化层

结构化层负责商品、活动、门店的结构化检索能力。
它的核心职责是：

* 提供统一的搜索候选召回能力
* 提供筛选、过滤、排序能力
* 提供面向 Agent 的标准查询接口
* 屏蔽底层业务系统复杂性

结构化层内部包括：

#### 4.1.1 快照表层

例如：

* `product_search_snapshot`
* `event_search_snapshot`
* `store_search_snapshot`

快照表不是业务真相表，而是为搜索场景构建的**查询投影层 / 读模型**。
它负责把商品、活动、门店相关的多源数据做扁平化、冗余化和派生计算，形成适合搜索和查询的统一结构。

#### 4.1.2 搜索索引层

例如：

* `product_search_v1`
* `event_search_v1`
* `store_search_v1`

这一层通常由 Elasticsearch / OpenSearch 承载，主要负责：

* 关键词检索
* 标题 / 描述全文搜索
* 标签召回
* 类目召回
* 地理检索
* 综合排序

#### 4.1.3 查询服务层

例如：

* `search_products`
* `search_events`
* `search_stores`

这些服务是结构化层对外暴露的标准能力入口。
Agent 不应直接访问底层表，而应通过结构化查询服务获取商品、活动、门店候选结果。

#### 4.1.4 同步构建机制

结构化层还需要有配套的数据构建与更新机制，包括：

* 事件驱动同步
* 定时补偿任务
* 索引重建
* 幂等与乱序处理
* 状态口径统一计算

这一机制负责将业务系统中的变化同步到快照表与 ES 索引。

---

### 4.2 知识层

知识层负责承载说明型、规则型、文档型信息。
它主要回答“是什么”“为什么”“怎么用”“有什么规则”这类问题。

知识层典型内容包括：

* 商品说明
* 活动说明
* 门店介绍
* 退款规则
* 使用规则
* FAQ
* 攻略内容
* 注意事项

知识层内部通常包括：

* 文档加载与解析
* chunk 切分
* embedding
* 向量存储
* retriever

LangChain 对 retrieval、retriever、vector store 有统一抽象，适合在这一层接入知识文档检索能力。

需要注意的是：

**LangChain 提供的是知识检索框架与抽象，不会自动替代业务知识库建设本身。**
知识内容、元数据、版本、生效状态、来源追踪，仍需要业务侧设计。

---

### 4.3 实时层

实时层负责提供在线实时确认能力。
它主要回答“现在是否可用”这类问题。

典型能力包括：

* 实时库存
* 实时价格
* 门店营业状态
* 活动余票
* 当日可预约状态

实时层不负责候选召回，也不负责知识解释。
它负责在用户已经有明确对象后，对最终状态做确认。

---

## 5. LangChain / LangGraph Agent 编排层

这一层是结合 LangChain 后最需要明确新增的一层。

这一层不负责存储业务数据，也不负责建立搜索索引。
它负责的是：

1. 理解用户自然语言输入
2. 提取结构化查询参数
3. 根据问题类型选择合适工具
4. 串联结构化层、知识层、实时层能力
5. 整合结果并生成最终回答

LangChain 官方支持 tools、structured output、agents 等能力；LangGraph 更适合构建多步骤、有状态、可控制的 agent workflow。

这一层建议拆成三个模块：

### 5.1 Query Parser

负责把用户自然语言转成结构化参数。
例如把：

> 推荐上海适合亲子的玩具，200 元以内

解析成：

```json
{
  "intent": "search_products",
  "keyword": "玩具",
  "city": "上海",
  "tags": ["亲子"],
  "max_price": 200,
  "sort_by": "recommend"
}
```

这一部分建议使用 LangChain 的 structured output 能力实现，以保证参数提取结果稳定、可控。

---

### 5.2 Tool Router

负责根据用户问题决定调用哪个能力层。

例如：

* “推荐几个适合送礼的东西”
  -> 调用 `search_products`

* “这个套餐包含什么”
  -> 调用知识层 retriever

* “这个活动今天还有票吗”
  -> 调用实时层接口

* “推荐几个适合亲子的活动，并告诉我哪个更适合 5 岁孩子”
  -> 先调用 `search_events`，再调用知识层补充说明

也就是说，这一层是整个系统的智能调度中心。

---

### 5.3 Response Composer

负责整合多工具返回结果，并生成用户最终可读的回答。

例如：

* 把结构化层返回的候选商品组织成推荐列表
* 把知识层返回的规则说明整理成自然语言解释
* 把实时层返回的库存 / 价格状态整合进最终回答
* 在必要时附带解释、原因和推荐依据

---

## 6. 用户交互层

用户交互层负责接收和展示最终结果。
典型入口包括：

* App
* 小程序
* Web 搜索页
* Chat UI
* 智能导购入口

这一层不直接与业务数据源交互，也不直接查询知识库或数据库。
它通过 LangChain / LangGraph Agent 编排层获取最终结果。

---

## 7. 调用链路说明

### 7.1 商品搜索类问题

例如用户说：

> 我想买一个漂亮的东西

系统处理流程为：

1. Agent 识别为商品推荐 / 搜索意图
2. Query Parser 提取 `keyword=漂亮`
3. Tool Router 调用 `search_products`
4. 结构化层完成候选召回与排序
5. Response Composer 组织商品推荐结果
6. 如用户继续问规则或用途，再调用知识层
7. 如用户确认购买，再调用实时层检查库存与价格

---

### 7.2 规则解释类问题

例如用户说：

> 这个活动能不能退？

系统处理流程为：

1. Agent 识别为规则问答意图
2. Tool Router 调用知识层 retriever
3. 知识层返回退款规则相关片段
4. Response Composer 生成自然语言解释

---

### 7.3 实时确认类问题

例如用户说：

> 这个商品现在还有货吗？

系统处理流程为：

1. Agent 识别为实时确认意图
2. Tool Router 调用实时库存接口
3. 实时层返回库存状态
4. Response Composer 生成最终回答

---

## 8. 结合 LangChain 后的边界说明

结合 LangChain 之后，需要明确以下边界：

### 8.1 LangChain 不替代结构化层

LangChain 可以调用数据库、API 和工具，但不会自动替你建设：

* 搜索快照表
* ES mapping
* 多路召回策略
* 状态口径
* 数据同步机制

因此，结构化层仍然是独立的业务搜索底座。

### 8.2 LangChain 不替代知识内容建设

LangChain 支持 retriever 和 vector store，但不会自动生成你的业务知识内容。
规则文档、FAQ、攻略、说明文档仍然需要自行建设与维护。

### 8.3 LangChain 最适合承担编排职责

LangChain / LangGraph 最适合放在：

* 意图识别
* 参数抽取
* 工具调用
* 多工具协同
* 回答生成

这一层，而不是放在底层数据存储或搜索索引层。

---

## 9. 最终推荐架构

最终推荐将系统设计为：

```text
用户
-> LangChain / LangGraph Agent 编排层
   -> Query Parser
   -> Tool Router
   -> Response Composer

底层能力层
-> 结构化层
   -> 快照表
   -> ES / OpenSearch
   -> 查询服务
-> 知识层
   -> 文档库
   -> 向量库
   -> Retriever
-> 实时层
   -> 库存接口
   -> 价格接口
   -> 营业状态接口

业务数据源层
-> 商品中心
-> 活动中心
-> 门店中心
-> 类目系统
-> 标签系统
-> 价格系统
-> 库存系统
-> 规则文档
```

---

## 10. 结论

结合 LangChain 后，总体架构不应推翻原有“结构化层 + 知识层 + 实时层”的分层思路。
正确的做法是：

**保留原有底层能力架构，在其上新增 LangChain / LangGraph Agent 编排层。**

这样做的好处是：

* 保留业务搜索底座的可控性
* 保留知识检索能力的独立性
* 保留实时接口的准确性
* 同时利用 LangChain / LangGraph 在 tool calling、structured output、retrieval 和 agent orchestration 上的优势。
