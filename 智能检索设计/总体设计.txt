# Agent + 结构化搜索 + 知识检索 + 实时查询

# 总设计文档（简约版）

## 1. 背景与目标

本方案的目标不是做一个纯聊天机器人，而是：

**用 Agent 把已有业务能力编排起来，形成自然语言交互入口。**

系统面向的典型问题包括：

* 商品、活动、门店搜索与推荐
* FAQ、规则、说明问答
* 搜索结果的实时确认
* 多来源结果融合与生成回答
* 多轮对话下的条件补充与收敛

---

## 2. 核心设计原则

### 2.1 大模型不替代原有业务系统

* 搜索仍由结构化层负责
* 规则与说明仍由知识层负责
* 库存、价格、营业状态等仍由实时层负责

### 2.2 大模型负责理解、编排、融合和生成

LLM 主要负责：

* 理解用户意图
* 提取和规范化参数
* 决定调用路径
* 融合多来源结果
* 生成自然语言回答

### 2.3 用户显式条件优先

建议优先级：

```text id="ovr_doc_01"
用户显式条件 > 上下文记忆 > 模型推断条件 > 系统默认条件
```

### 2.4 执行参数尽量收敛到 ID

生产执行参数应优先使用标准 ID，而不是 name。

---

## 3. 总体分层架构

系统建议分为七层：

1. 用户交互层
2. Agent 编排层
3. 结构化层
4. 知识层
5. 实时层
6. 数据同步与索引构建层
7. 业务数据源层

### 3.1 用户交互层

负责接收用户问题并展示最终结果。

### 3.2 Agent 编排层

负责：

* 意图识别
* 参数标准化
* 工具路由
* 多工具编排
* 结果融合
* 多轮会话状态维护

### 3.3 结构化层

负责：

* 商品 / 活动 / 门店搜索
* 精确过滤
* 排序
* TopN 候选生成

### 3.4 知识层

负责：

* FAQ
* 规则
* 说明
* 攻略
* 注意事项

### 3.5 实时层

负责：

* 当前库存
* 当前价格
* 当前营业状态
* 当前余票
* 当前可预约状态

### 3.6 数据同步与索引构建层

负责：

* 主表到快照表同步
* 快照表到 ES 同步
* 知识文档到知识索引同步
* 全量 / 增量 / 补偿
* alias 切换

### 3.7 业务数据源层

负责业务主数据和实时业务真相。

---

## 4. 技术选型原则

本方案不是“LangChain 或 LangGraph 二选一”，而是分层使用：

### 4.1 LangChain 负责组件接入层

用于：

* 模型接入
* tools 封装
* retrievers 封装
* prompt
* output parser

### 4.2 LangGraph 负责编排运行时层

用于：

* 节点编排
* Session State
* Tool Router
* 串行 / 并行控制
* checkpoint / 恢复
* tracing / debugging

### 4.3 不建议由 LangChain / LangGraph 主要承载的部分

* 结构化搜索底座
* 实时服务底座
* 数据同步链路
* 业务规则本体

## 5. Agent 运行时主流程

建议的核心节点如下：

* `parse_intent`
* `normalize_params`
* `route_tools`
* `execute_structured_search`
* `retrieve_knowledge`
* `query_realtime`
* `compose_response`

总体主流程：

```text id="ovr_doc_02"
用户输入
-> parse_intent
-> normalize_params
-> route_tools
-> 执行结构化搜索 / 知识检索 / 实时查询
-> compose_response
-> 最终回答
```

---

## 6. Session State 设计

Session State 是 Agent 的运行时状态容器，用于保存：

* 消息历史
* 会话上下文
* 意图识别结果
* 标准化参数
* 工具执行计划
* 搜索结果
* 知识结果
* 实时结果
* 最终答案
* 调试轨迹

建议核心字段包括：

* `messages`
* `user_profile`
* `session_context`
* `parsed_intent`
* `normalized_params`
* `tool_plan`
* `search_results`
* `retrieved_docs`
* `realtime_results`
* `final_answer`
* `debug_trace`

---

## 7. Tool Router 设计

Tool Router 是工具路由与调度层，负责：

* 决定调用哪个工具
* 决定单工具还是多工具
* 决定串行还是并行
* 决定调用顺序
* 决定是否需要澄清
* 决定失败后的降级路径

它不负责：

* 原始自然语言理解
* 最终回答生成
* 具体工具内部实现

典型路由模式包括：

* 单工具结构化搜索
* 单工具知识检索
* 搜索后实时确认
* 搜索后知识补充
* 搜索后并行知识 + 实时
* 澄清优先

---

## 8. 三类能力层的职责边界

## 8.1 结构化层

负责：

* 搜索召回
* 过滤
* 排序
* TopN 候选生成

不负责：

* 长文本规则说明
* 实时状态判断

## 8.2 知识层

负责：

* FAQ
* 规则
* 使用说明
* 注意事项
* 攻略与导览

不负责：

* 大规模结构化召回
* 实时状态判断

## 8.3 实时层

负责：

* 当前库存
* 当前价格
* 当前营业状态
* 当前余票
* 当前预约可用性

不负责：

* 海量召回
* 排序推荐
* 长文本解释

### 8.4 多来源冲突优先级

建议统一为：

```text id="ovr_doc_03"
实时层 > 结构化层快照 > 知识层文档 > LLM 推断
```

## 9. Response Composer 设计

Response Composer 负责把多来源结果融合为最终回答。

核心职责包括：

* 统一融合结构化、知识、实时结果
* 处理冲突
* 生成推荐理由
* 生成 disclaimer
* 输出标准化 answer schema
* 生成下一轮引导

推荐输出结构包括：

* `answer_text`
* `cards`
* `disclaimers`
* `citations`
* `next_actions`

## 10. 数据同步与索引构建设计

### 10.1 结构化主链路

建议采用：

```text id="ovr_doc_04"
业务主表 / 配置表
-> 快照构建
-> 搜索快照表
-> ES / OpenSearch
```

### 10.2 知识主链路

建议采用：

```text id="ovr_doc_05"
知识源文档
-> 文档标准化
-> chunk 切分
-> metadata 补齐
-> 知识索引 / 向量索引
```

### 10.3 实时层

实时层不通过同步索引保障当前真相，而是在查询时按需调用。

### 10.4 `searchable_status`

建议在快照层统一计算，不建议放到 ES 查询阶段动态拼装。

### 10.5 同步机制

必须同时具备：

* 全量同步
* 增量同步
* 补偿同步

### 10.6 索引治理

ES / OpenSearch 重建建议通过 alias 切换完成。

## 14. 异常处理与降级策略

### 14.1 LLM 层降级

| 异常场景 | 降级策略 |
|---------|---------|
| LLM 调用超时（>3s） | 中断当前调用，使用模板回答 + 原始搜索结果直出 |
| LLM 返回格式异常 | 重试一次，仍失败则走规则引擎兜底 |
| LLM 服务完全不可用 | 切换备用模型；若备用也不可用，降级为纯结构化搜索 + 模板拼装 |
| Token 配额耗尽 | 返回排队提示，引导用户稍后重试 |

### 14.2 结构化层降级

| 异常场景 | 降级策略 |
|---------|---------|
| ES 查询超时 | 缩小查询范围（减少条件、缩短时间窗口）重试一次 |
| ES 查询无结果 | 逐步放宽条件（去掉最弱约束）；仍无结果则返回热门推荐 + 提示"未找到精确匹配" |
| ES 集群不可用 | 切换到备用集群 / 只读副本；完全不可用则返回缓存的热门数据 |

### 14.3 知识层降级

| 异常场景 | 降级策略 |
|---------|---------|
| 检索 confidence < 阈值（如 0.6） | 不返回该知识片段，回答中标注"未找到明确说明，建议咨询人工客服" |
| 向量索引不可用 | 降级为关键词检索（BM25） |
| 检索到多条冲突知识 | 取最新版本；若无法判断版本，全部列出并标注"以下信息可能存在差异" |

### 14.4 实时层降级

| 异常场景 | 降级策略 |
|---------|---------|
| 实时接口超时 / 不可用 | 使用快照层数据兜底，回答中标注"数据更新截止到 [快照时间]，请以实际为准" |
| 实时接口返回异常数据 | 丢弃该数据，不展示，不影响其他来源结果 |

### 14.5 全链路降级原则

```text id="ovr_doc_13"
1. 能回答就回答，不能精确回答就模糊回答 + 标注不确定性
2. 降级后必须在回答中明确告知用户数据可能不准确
3. 所有降级事件必须记录到 debug_trace，触发告警
4. 禁止在降级场景下返回"编造"的数据
```

## 15. 安全与限流设计

### 15.1 Prompt Injection 防护

* 用户输入在送入 LLM 前必须经过预处理：
  * 过滤已知注入模式（如 "ignore previous instructions"、"system prompt" 等）
  * 使用 input / output guardrail，对输入和输出分别做安全检测
* System Prompt 与用户输入严格隔离，使用 `<<< USER_INPUT >>>` 等标记界定边界
* 定期维护注入模式黑名单，结合线上日志更新

### 15.2 敏感信息过滤

* 回答输出前经过敏感信息检测：
  * 不泄露内部表名、字段名、接口地址
  * 不泄露其他用户的个人信息
  * 不泄露内部定价策略、成本数据
* 搜索结果中的内部字段（如 internal_id、cost_price）在 compose_response 阶段过滤，不传递到最终回答

### 15.3 限流与成本控制

| 维度 | 策略 |
|------|------|
| 单用户 QPS | 限制 2 QPS，突发允许 5 |
| 单用户日调用量 | 上限 200 次 / 天 |
| 单次请求 Token 上限 | Input 2000 tokens，Output 1000 tokens |
| 全局 QPS | 根据 LLM 配额设定，预留 20% buffer |
| 成本预警 | 日消耗达到预算 80% 时告警，达到 100% 时降级为缓存回答 |

### 15.4 审计日志

每次请求必须记录：

* `request_id`
* `user_id`
* `timestamp`
* `raw_input`（用户原始输入）
* `parsed_intent`
* `tool_calls`（调用了哪些工具、参数、耗时、结果摘要）
* `final_answer`
* `is_degraded`（是否降级）
* `risk_level`（高风险场景标记）

审计日志保留期限不少于 90 天，高风险场景日志保留不少于 180 天。

## 16. 性能预算与超时设计

### 16.1 端到端 SLA 目标

| 指标 | 目标 |
|------|------|
| P50 | < 2s |
| P95 | < 4s |
| P99 | < 6s |
| 最大超时 | 8s（超时后强制返回降级回答） |

### 16.2 各节点超时预算分配

```text id="ovr_doc_14"
parse_intent:             500ms（含 LLM 调用）
normalize_params:         200ms
route_tools:              300ms（含 LLM 调用）
execute_structured_search: 800ms
retrieve_knowledge:        800ms
query_realtime:            500ms
compose_response:          800ms（含 LLM 调用）

串行链路总预算:           3900ms
并行可压缩到:             约 2500ms
```

### 16.3 并行与串行策略

| 场景 | 执行方式 | 触发条件 |
|------|---------|---------|
| 搜索 + 知识 | 并行 | route_tools 判定两者独立时 |
| 搜索 + 实时 | 串行 | 实时查询依赖搜索结果的 ID |
| 知识 + 实时 | 并行 | 两者无依赖关系时 |
| 搜索 + 知识 + 实时 | 搜索先行，知识和实时并行 | 实时依赖搜索结果，知识不依赖 |

### 16.4 缓存策略

* 意图相同 + 参数相同的请求，结构化搜索结果缓存 5 分钟
* 知识检索结果缓存 30 分钟（知识变更频率低）
* 实时数据不缓存
* 最终回答不缓存（因为包含实时数据）

---

## 17. 多轮对话上下文管理

### 17.1 上下文窗口管理

* 消息历史最多保留最近 10 轮（20 条消息）
* 超过 10 轮时，使用 LLM 对早期对话生成摘要，替换原始消息
* 摘要格式：`{summary: "...", key_params: {...}, resolved_entities: [...]}`
* 每次传入 LLM 的上下文结构：`[system_prompt] + [summary] + [recent_messages]`

### 17.2 意图切换处理

| 场景 | 处理策略 |
|------|---------|
| 用户明确切换话题（如"换个问题"） | 清空 parsed_intent、normalized_params、tool_plan，保留 user_profile |
| 用户隐式切换（前一轮问商品，这一轮问门店） | parse_intent 检测到意图不同时，清空上一轮的搜索参数，保留通用参数（如 location） |
| 用户在原意图上追加条件（如"再便宜一点"） | 保留上一轮参数，merge 新条件 |
| 用户否定上一轮结果（如"不要这个"） | 将被否定的结果 ID 加入排除列表 |

### 17.3 Session 生命周期

| 事件 | 处理 |
|------|------|
| 新会话开始 | 创建新 Session State，初始化 user_profile |
| 用户 30 分钟无操作 | Session 过期，标记为 inactive |
| 用户在过期后发消息 | 创建新 Session，但尝试加载上一次 Session 的 user_profile |
| 用户主动说"重新开始" | 清空当前 Session，重新初始化 |

## 18. 测试与上线策略

### 18.1 测试体系

| 测试类型 | 覆盖范围 | 执行频率 |
|---------|---------|---------|
| 单元测试 | 各节点的输入输出、参数标准化逻辑、路由规则 | 每次提交 |
| 集成测试 | 完整链路（mock LLM + 真实 ES） | 每日构建 |
| Golden Test | 核心场景的标准问答对（50+ case） | 每次 prompt / 模型变更前 |
| 压力测试 | 并发场景下的性能与稳定性 | 上线前 |
| 对抗测试 | Prompt injection、恶意输入、边界输入 | 每次 prompt 变更前 |

### 18.2 Golden Test Case 分类

必须覆盖的核心场景：

* 单条件商品搜索
* 多条件组合搜索
* 无结果场景
* FAQ 精确命中
* 知识模糊匹配
* 搜索 + 实时确认
* 多轮追问与条件追加
* 意图切换
* 高风险场景（退款、价格、营业状态）
* 澄清引导

### 18.3 灰度方案

```text id="ovr_doc_15"
阶段一：内部测试
  范围：内部员工
  时长：1 周
  目标：功能验证 + 问题收集

阶段二：小流量灰度
  范围：5% 线上用户（按 user_id 尾号分桶）
  时长：1-2 周
  目标：线上稳定性验证 + 性能指标达标

阶段三：扩大灰度
  范围：20% -> 50% -> 100%
  时长：每阶段观察 3-5 天
  目标：全量验证 + 监控告警覆盖

回滚条件：
  - 错误率 > 5%
  - P95 > 6s
  - 高风险场景出现事实性错误
  - 用户投诉率突增
```

### 18.4 A/B 测试

* 对照组：原有搜索 / 客服入口
* 实验组：Agent 入口
* 核心对比指标：
  * 用户问题解决率
  * 平均交互轮次
  * 用户满意度评分
  * 转人工率

### 18.5 高风险场景人工审核

以下场景的回答在灰度期间需要异步人工抽检：

* 涉及退款规则的回答
* 涉及当前价格的回答
* 涉及年龄 / 入场限制的回答
* 涉及营业状态的回答

抽检比例：灰度期间 100%，全量后 10%。

## 19. 知识层更新时效性

### 19.1 知识更新 SLA

| 知识类型 | 更新 SLA | 说明 |
|---------|---------|------|
| FAQ | 4 小时内可检索 | 日常更新，批量同步 |
| 业务规则（退款、入场限制等） | 1 小时内可检索 | 高优先级，触发即时同步 |
| 攻略 / 导览 | 24 小时内可检索 | 低频更新，随全量同步 |
| 临时公告（闭园、临时调价等） | 30 分钟内可检索 | 走紧急通道，手动触发即时同步 |

### 19.2 知识冲突处理

```text id="ovr_doc_16"
冲突判定：同一主题存在多条知识片段且内容矛盾
处理规则：
  1. 优先使用 effective_date 最新的版本
  2. 若无 effective_date，使用 updated_at 最新的版本
  3. 若仍无法判定，全部返回并标注"以下信息可能存在更新，请以最新公告为准"
  4. 冲突事件记录到治理日志，触发人工审核
```

### 19.3 知识版本管理

* 每条知识文档必须包含元数据：`doc_id`、`version`、`effective_date`、`expire_date`、`updated_at`、`status`
* 知识状态流转：`draft -> active -> deprecated -> archived`
* 过期知识（`expire_date < now`）自动标记为 `deprecated`，从索引中移除
* 归档知识保留 180 天，可用于审计追溯，不参与检索

## 20. 边界声明

### 20.1 本系统负责

* 商品、活动、门店的自然语言搜索与推荐
* FAQ、规则、说明的问答
* 搜索结果的实时状态确认
* 多来源结果的融合回答
* 多轮对话的条件收敛与澄清

### 20.2 本系统不负责

* **不做交易下单** — 仅提供查询与推荐，不执行购买、预约等写操作
* **不做用户身份认证** — 依赖上游系统传入的 user_id / token，不自行管理登录态
* **不做跨租户数据访问** — 严格按租户隔离，不支持跨租户查询
* **不替代人工客服** — 复杂投诉、纠纷处理、情感安抚等场景引导转人工
* **不做个性化营销推送** — 不主动触达用户，仅在用户发起查询时响应
* **不承诺回答 100% 准确** — 所有涉及价格、库存、营业状态的回答均附 disclaimer

### 20.3 系统扩展预留

以下能力不在当前版本范围，但架构设计上预留扩展点：

* 个性化推荐（基于用户历史偏好）
* 多语言支持
* 语音交互接入
* 下单等写操作（需独立安全评审后接入）

## 21. 最终结论

本方案建议采用如下总体架构：

```text id="ovr_doc_12"
LangChain 作为模型、tool、retriever 集成层
+ LangGraph 作为 Agent 编排与状态运行时
+ 结构化搜索层独立实现
+ 知识层独立实现
+ 实时服务层独立实现
+ 数据同步与索引构建层独立实现
```

其核心价值在于：

1. 保留原有业务能力边界
2. 用 Agent 形成统一自然语言入口
3. 让系统既能“找得到”，也能“说得清”，还能“查得准”
4. 通过状态驱动实现多轮、多工具、可恢复的编排
5. 通过同步、监控、评估与治理保证长期可用性与稳定性
