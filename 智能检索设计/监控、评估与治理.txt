# 《监控、评估与治理 V1》

这份文档的目标，是把前面已经完成的：

* Tool Router
* Session State
* 知识层
* 实时层
* Response Composer
* 同步链路

真正变成一套**可上线、可观测、可评估、可治理**的系统。

因为 Agent 架构最容易出的问题，不是“能不能跑起来”，而是：

* 跑起来之后不知道哪里错了
* 错了之后不知道怎么定位
* 改了之后不知道有没有变好
* 上线后没有稳定的质量护栏
* 幻觉、旧数据、错误路由、错误融合混在一起

所以这份文档的核心目标是：

**建立从数据、节点、链路、结果到运营反馈的全链路治理体系。**

---

# 一、监控、评估与治理 V1

## 1. 文档目标

本文档定义 Agent 系统在生产环境中的：

1. 运行监控体系
2. 质量评估体系
3. 数据治理体系
4. 幻觉与事实错误治理
5. 线上抽检与回放机制
6. 发布与变更治理
7. 问题定位与归因框架

本文档解决的问题包括：

* 怎么知道系统是否健康
* 怎么知道回答质量是否在变差
* 怎么知道错在路由、检索、实时还是回答融合
* 怎么控制高风险问题
* 怎么做版本发布、灰度和回滚
* 怎么形成长期评估闭环

---

## 2. 设计原则

## 2.1 可观测优先

每个关键节点都必须产出可观测信息，不能只看最终用户反馈。

---

## 2.2 评估分层，不做“单指标迷信”

不能只看一个总满意度。
应分层看：

* 意图识别
* 参数标准化
* 路由
* 检索
* 实时查询
* 回答融合
* 用户反馈

---

## 2.3 事实类问题高于表达类问题

一个回答即使文案一般，只要事实正确，通常还可接受。
但如果事实错了，再流畅都不行。

---

## 2.4 高风险场景单独治理

例如：

* 退款规则
* 当前可订状态
* 当前营业状态
* 当前价格
* 年龄 / 入场限制

这类内容必须有更严格的治理策略。

---

## 2.5 线上指标与离线评估必须联动

离线分数高不代表线上稳定。
线上点击高也不代表事实正确。
两者必须结合。

---

## 2.6 治理对象不仅是模型，也包括数据和流程

Agent 出错不一定是模型错，可能是：

* 快照过期
* 知识文档旧版本命中
* Tool Router 选错路径
* 实时查询超时
* Response Composer 静默覆盖冲突

所以治理对象必须覆盖整个系统。

---

## 3. 治理范围分层

建议将治理拆成五层：

```text id="gov_layers_01"
L1 数据层治理
L2 节点层治理
L3 链路层治理
L4 最终回答治理
L5 发布与运营治理
```

---

## 4. 运行监控体系

先定义“系统是否健康”。

## 4.1 端到端运行指标

建议监控以下全局指标：

* 请求总量
* 成功响应率
* 平均响应时延
* P95 / P99 响应时延
* 无结果率
* 部分结果率
* 降级率
* 澄清率
* 失败率

这些指标回答的是：

* 系统有没有在正常服务
* 用户有没有明显感知到性能或结果问题

---

## 4.2 节点级运行指标

每个关键节点都建议监控：

* 调用次数
* 成功率
* 失败率
* 平均耗时
* P95 / P99 耗时
* 超时率
* 空结果率
* 降级率

适用节点包括：

* `parse_intent`
* `normalize_params`
* `route_tools`
* `execute_structured_search`
* `retrieve_knowledge`
* `query_realtime`
* `compose_response`

---

## 4.3 工具级运行指标

每类工具也要单独监控。

### 结构化工具

* `search_products`
* `search_events`
* `search_stores`

建议监控：

* QPS
* 查询耗时
* 空结果率
* 过滤后结果数分布
* ES 错误率

### 知识检索工具

* topK 返回量
* rerank 耗时
* 文档命中率
* 空命中率
* 过期文档命中率

### 实时工具

* 成功率
* 超时率
* 部分失败率
* 实体平均查询数
* 限流率

---

## 4.4 状态流转监控

LangGraph 类系统最好增加状态流转监控，例如：

* 各 `task_type` 占比
* 各 `plan_type` 占比
* `execution_mode` 分布
* `clarification_required` 比例
* `mixed_serial_parallel` 占比
* fallback 触发次数

这些指标很有价值，因为它们能直接反映：

* 用户真实请求结构
* Router 是否过度依赖澄清
* 系统是否越来越重实时
* 是否出现异常的路由漂移

---

## 5. 质量评估体系

运行正常不代表回答质量好。
这里单独定义质量评估。

## 5.1 评估分层

建议按以下层级评估：

1. 意图识别质量
2. 参数标准化质量
3. 路由质量
4. 结构化搜索质量
5. 知识检索质量
6. 实时确认质量
7. 回答融合质量
8. 最终回答质量

---

## 5.2 意图识别评估

建议评估：

* `task_type` 准确率
* `need_realtime` 判断准确率
* `need_explanation` 判断准确率
* `need_recommendation` 判断准确率

### 常见错误类型

* 搜索型误判成 FAQ
* FAQ 误判成搜索
* 强实时问题未识别为实时
* 推荐型问题未识别为推荐

---

## 5.3 参数标准化评估

建议评估：

* 槽位抽取准确率
* ID 映射准确率
* 显式条件保留率
* 默认条件注入正确率
* `missing_required_slots` 识别正确率

### 重点检查项

* 用户显式条件是否被覆盖
* name -> id 是否映射正确
* 时间解析是否正确
* 地域解析是否正确
* 类目 / 标签映射是否正确

---

## 5.4 Tool Router 评估

建议评估：

* 路由正确率
* 计划类型正确率
* 实时触发正确率
* 知识补充触发正确率
* 澄清触发准确率
* 降级策略触发合理率

### 重点错误类型

* 应该查实时却没查
* 不需要查实时却查了
* 应该查知识却没有查
* 应该澄清却直接执行
* 能直接执行却过度澄清

---

## 5.5 结构化搜索评估

建议评估：

* top1 / top3 / top10 相关性
* 过滤正确率
* 排序符合业务预期比例
* 无结果误判率
* 核心字段展示完整率

---

## 5.6 知识检索评估

建议评估：

* top1 命中率
* top3 命中率
* 错版本命中率
* 过期文档命中率
* 错实体命中率
* FAQ 直接命中率

高风险 doc_type 建议单独测：

* `refund_rule`
* `reservation_rule`
* `usage_notice`

---

## 5.7 实时层评估

建议评估：

* 强实时问题实时触发率
* 实时结果覆盖率
* 实时结果正确率
* 部分失败对最终答案影响率
* 快照与实时冲突发现率

---

## 5.8 Response Composer 评估

建议评估：

* 冲突处理正确率
* 推荐理由真实性
* disclaimer 触发正确率
* final answer schema 完整率
* no-result 表达正确率
* partial-result 表达正确率

---

## 5.9 最终回答评估

建议综合评估：

* 事实正确率
* 用户可读性
* 相关性
* 完整性
* 可操作性
* 下一步引导合理性

---

## 6. 评估集设计

没有稳定评估集，就很难做版本比较。

## 6.1 评估集分桶

建议至少按以下场景建评估桶：

### A. 商品 / 活动 / 门店推荐

例如：

* 亲子
* 雨天
* 周末
* 低价
* 附近

### B. FAQ / 规则问答

例如：

* 退款
* 停车
* 预约
* 年龄限制
* 营业说明

### C. 搜索 + 实时确认

例如：

* 今天还能订吗
* 当前营业吗
* 还有票吗

### D. 混合型问题

例如：

* 推荐几个现在还能去的地方，并告诉我注意事项

### E. 多轮澄清问题

例如：

* “附近有什么好玩的”
* “这个能退吗”
* “那换便宜一点的”

---

## 6.2 评估集标签设计

每条样本建议带：

* `task_type`
* `risk_level`
* `need_realtime`
* `need_explanation`
* `need_clarification`
* `entity_scope`
* `expected_tools`
* `expected_constraints`
* `expected_answer_type`

这样后续可以按桶分析。

---

## 6.3 高风险样本池

建议单独维护高风险样本池。
例如：

* 退款 / 改签规则
* 当前可订状态
* 当前营业状态
* 当前价格
* 儿童 / 身高 / 入场限制
* 节假日特殊规则

这些样本应在每次版本发布前强制回归。

---

## 7. 幻觉与事实错误治理

这是 Agent 系统必须单独面对的一类问题。

## 7.1 幻觉分类

建议将幻觉 / 事实错误分为：

### 7.1.1 无证据推荐理由

例如：

* “这个更适合低龄儿童”
  但没有任何结构化或知识证据支撑。

---

### 7.1.2 错误状态断言

例如：

* “现在还能订”
* “当前正在营业”

但没有实时结果或实时失败。

---

### 7.1.3 错误规则归纳

例如：

* 把规则文档意思说反了
* 把“24小时外可退”说成“24小时内可退”

---

### 7.1.4 静默覆盖冲突

例如：

* 实时与知识冲突，但回答里只给一个结论且不解释

---

## 7.2 幻觉治理原则

### 原则 1：高风险事实尽量结构化输出后再表达

例如：

* `bookable=true`
* `refund_rule_summary=...`

不要让 LLM 直接从混乱上下文自由发挥。

### 原则 2：强约束规则程序化

例如：

* 没有实时结果就不能输出“当前可订”
* 没有知识证据就不能输出“支持退款”

### 原则 3：高风险场景引用优先

例如：

* 规则结论尽量附 citation 或 traceable source

---

## 7.3 幻觉检测建议

建议增加以下检测：

* 推荐理由是否有 source support
* 强实时结论是否有实时结果支撑
* 规则型结论是否有知识证据支撑
* disclaimer 缺失检测
* 冲突静默覆盖检测

这些可以部分程序化完成。

---

## 8. 数据治理体系

很多线上问题其实是数据问题，不是 Agent 推理问题。

## 8.1 结构化数据治理

建议治理：

* 标题为空率
* 类目缺失率
* tag 缺失率
* `searchable_status` 异常率
* 城市 / 区域映射异常率
* 地理坐标异常率

---

## 8.2 知识数据治理

建议治理：

* `doc_type` 缺失率
* `entity_id` 缺失率
* 生效时间缺失率
* owner 缺失率
* 过期文档占比
* 多版本冲突率
* chunk 空文本率

---

## 8.3 实时数据治理

建议治理：

* 状态枚举非法值
* 价格异常值
* 返回时间戳缺失率
* 实体不存在率
* 服务返回字段不一致率

---

## 8.4 数据质量告警

建议为这些问题设定告警阈值：

* 高风险规则文档缺失率
* `enabled` 快照量异常波动
* 实时成功率异常下降
* 某主要类目结果量突降
* 某城市核心门店状态异常集中

---

## 9. 线上抽检机制

光看指标还不够，必须有人能看见真实回答质量。

## 9.1 抽检维度

建议按以下维度做线上抽检：

* 高频 query
* 高风险 query
* 新上线 query 桶
* 降级样本
* 无结果样本
* 高投诉样本
* 实时失败样本

---

## 9.2 抽检内容

每条样本建议展示：

* 用户原 query
* `parsed_intent`
* `normalized_params`
* `tool_plan`
* `search_results` 摘要
* `retrieved_docs` 摘要
* `realtime_results` 摘要
* `final_answer`
* `disclaimers`
* `debug_trace`

这样评审人员才能知道错在哪一层。

---

## 9.3 抽检标注项

建议人工抽检时至少标这些项：

* 意图是否正确
* 参数是否正确
* 路由是否合理
* 检索是否命中
* 实时是否该查
* 回答是否事实正确
* 推荐理由是否有依据
* disclaimer 是否足够
* 下一步引导是否自然

---

## 10. 回放与问题定位机制

Agent 系统一定要支持回放，不然线上问题很难定位。

## 10.1 为什么必须支持回放

用户反馈一句“你答错了”，如果没有回放能力，很难知道是：

* 解析错了
* 映射错了
* 路由错了
* 搜索没召回
* 实时没查到
* Composer 说错了

---

## 10.2 推荐回放对象

建议至少可回放：

* 原 query
* state checkpoints
* tool inputs / outputs 摘要
* 版本号信息
* prompt version
* parser version
* routing reason
* final answer object

---

## 10.3 问题归因框架

建议线上问题按以下路径归因：

### A. 数据问题

* 主表错
* 快照错
* ES 错
* 文档旧
* 实时服务错

### B. 理解问题

* task_type 错
* 槽位抽错
* 约束错

### C. 路由问题

* 工具选错
* 顺序错
* 漏查实时
* 漏查知识

### D. 融合问题

* 冲突没处理
* 推荐理由瞎编
* disclaimer 漏掉

### E. 表达问题

* 文案歧义
* 口语化过度
* 没回答用户核心问题

---

## 11. 发布治理与灰度机制

## 11.1 为什么需要发布治理

这套系统会频繁变更：

* prompt
* parser
* 路由规则
* 检索策略
* rerank
* 索引版本
* 文档 schema
* 状态字段逻辑

没有发布治理，很容易线上抖动。

---

## 11.2 建议治理对象

建议把以下都版本化：

* prompt version
* parser version
* router rule version
* snapshot schema version
* search index version
* kb index version
* composer template version

---

## 11.3 灰度策略建议

新版本建议先灰度到：

* 特定 query 桶
* 特定低风险场景
* 小流量用户
* 内部账号
* 回放样本集

重点观察：

* 成功率
* 时延
* 降级率
* 无结果率
* 事实正确率抽检
* 用户二次追问率

---

## 11.4 回滚机制

建议所有高风险变更都支持快速回滚，例如：

* Router 规则回滚
* prompt 回滚
* index alias 回滚
* kb 索引版本回滚
* composer 模板回滚

---

## 12. 线上运营反馈闭环

## 12.1 用户反馈入口

建议沉淀：

* 有帮助 / 没帮助
* 答非所问
* 信息过时
* 规则错误
* 当前状态不准
* 推荐不相关

这些反馈要能映射到治理标签。

---

## 12.2 投诉与人工接管样本

人工接管样本非常有价值。
建议沉淀为：

* 高风险失败样本池
* 回归测试池
* Router 规则修正规则集
* 高风险知识校正池

---

## 12.3 运营分析报表

建议定期出报表：

* 高频 query topN
* 高投诉 query topN
* 高降级 query topN
* 无结果 query topN
* 实时失败 topN
* 知识缺失 topN
* 路由争议样本 topN

这些会直接指导下一轮优化。

---

## 13. 建议指标总表

下面给一版精简指标框架。

## 13.1 系统健康类

* 澄清率

* 快照缺字段率
* 知识 metadata 缺失率
* 过期文档命中率

## 13.3 理解与路由类

* 意图准确率

* 路由准确率
* 实时触发准确率
* 澄清准确率

## 13.4 检索与查询类

* 搜索 top3 相关率
* 知识 top1 命中率

* 事实正确率

## 14. 推荐评审机制

建议建立固定节奏的评审：

### 每日看板

看系统健康与异常波动。

### 每周样本评审

看：

* 高频 query
* 高风险 query
* 失败 query
* 投诉 query

### 每次发布前回归

强制跑：

* 高风险样本池
* 主要场景评估集
* 新增规则样本
* 新索引抽检

## 17. 最终结论

监控、评估与治理在这套系统中的正确定位是：

**保证 Agent 不只是“能工作”，而是“能稳定工作、能持续变好、出了问题能定位”。**

它的核心价值在于：

1. 把复杂系统拆成可观测、可评估的层
2. 把“模型问题”与“数据问题”“路由问题”“融合问题”区分开
3. 让发布、灰度、回滚、回放成为标准动作
4. 让高风险场景有明确护栏
5. 形成持续优化闭环

到这里，你这套架构已经基本具备一份完整总体设计稿的主干了。
