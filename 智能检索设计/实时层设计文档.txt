# 《实时层正式设计 V1》

这份文档的目标，是把实时层从“查几个即时接口”收敛成一个边界清晰、成本可控、可降级的业务能力层。

---

# 一、实时层正式设计 V1

## 1. 文档目标

本文档定义 Agent 架构中实时层的职责、接口类型、调用时机、结果模型、降级策略和与结构化层、知识层的边界。

实时层的核心目标是：

**为 Agent 提供高时效、强状态型、当前时刻相关的业务事实。**

这些事实通常包括：

* 当前库存
* 当前价格
* 当前营业状态
* 当前余票
* 当前可预约状态
* 当前售卖状态
* 当前排队 / 满位 / 停止接待状态
* 当前场次可用情况

实时层解决的问题是：

* 现在还能买吗
* 今天还能订吗
* 当前还营业吗
* 当前有没有票
* 当前价格是多少
* 当前还能预约吗

它不解决的问题是：

* 海量对象召回
* 复杂排序推荐
* 长文本规则解释
* 业务主数据组织
* 历史规则说明

---

## 2. 设计定位

在整体架构中，实时层位于：

```text id="rt_arch_01"
结构化候选 / 已知实体
-> Agent 编排层
-> 实时查询
-> 返回当前状态事实
-> Response Composer 融合生成
```

实时层是一个**当前状态事实层**。
它本质上服务于“确认”而不是“发现”。

也就是说，它更适合回答：

* 候选里哪些当前可用
* 某个具体对象现在是什么状态

而不适合直接承担：

* 从全量对象中找谁最适合
* 对海量对象做大范围检索
* 用自然语言做大规模召回

---

## 3. 职责边界

## 3.1 实时层负责什么

实时层负责以下类型的即时状态查询。

### 3.1.1 库存类

例如：

* 商品当前库存
* 套餐当前可售数量
* 某 SKU 当前是否可买

---

### 3.1.2 价格类

例如：

* 当前售价
* 当前活动价
* 当前折扣价
* 当前票档价格
* 当前时间窗对应价格

---

### 3.1.3 营业 / 接待类

例如：

* 门店当前是否营业
* 当前是否停止入场
* 当前是否满位
* 当前是否暂停接待

---

### 3.1.4 余票 / 场次类

例如：

* 今天某场次是否还有票
* 明天某时间段是否还能订
* 当前某活动是否还有名额

---

### 3.1.5 预约可用性类

例如：

* 当前是否可预约
* 某时间窗是否可约
* 某日期是否仍有可预约资源

---

## 3.2 实时层不负责什么

### 3.2.1 不负责全量搜索召回

例如：

* 搜所有活动
* 按类目和城市过滤
* 按评分和热度排序

这些属于结构化层。

---

### 3.2.2 不负责规则解释

例如：

* 退款规则
* 使用规则
* 适合人群
* 入场限制

这些属于知识层。

---

### 3.2.3 不负责长期静态说明

例如：

* 门店介绍
* 产品亮点
* 商圈玩法说明

这些属于知识层或结构化层快照字段。

---

## 4. 与结构化层、知识层的边界

## 4.1 与结构化层的边界

### 结构化层负责

* 候选召回
* 过滤
* 排序
* TopN 生成
* 基于快照的可搜索事实

### 实时层负责

* 对候选对象做当前状态确认
* 对具体实体做即时事实补充

### 典型正确链路

```text id="rt_boundary_01"
search_events
-> topN candidate ids
-> query_realtime_availability
-> compose_response
```

而不是：

```text id="rt_boundary_02"
query_realtime_availability over all events
-> try to sort and recommend
```

后者通常成本高、稳定性差，也不适合生产。

---

## 4.2 与知识层的边界

### 知识层负责

* 一般说明
* 规则文本
* 常规营业说明
* 预约规则说明
* 价格规则说明

### 实时层负责

* 当前状态
* 当前是否可买
* 当前是否营业
* 当前是否还有余量
* 当前价格结果

---

## 4.3 冲突优先级

建议统一采用：

```text id="rt_priority_01"
实时层 > 结构化层快照 > 知识层文档 > LLM 推断
```

### 示例 1

知识文档写着：

“门店通常营业至 22:00。”

但实时接口返回：

`business_status = paused`

则最终回答应以实时层为准，并说明：

“常规营业说明是到 22:00，但当前实时状态显示暂停营业。”

### 示例 2

结构化快照里仍显示“可售”，但实时库存已为 0。
则以实时层为准。

---

## 5. 实时能力分类设计

为了避免“实时层”过于抽象，建议按能力分类。

## 5.1 商品实时能力

适用于 `product` / `sku` / `package` 等实体。

建议能力包括：

* `query_realtime_inventory`
* `query_realtime_price`
* `query_realtime_sell_status`

---

## 5.2 活动实时能力

适用于 `event` / `session` / `ticket_type` 等实体。

建议能力包括：

* `query_realtime_availability`
* `query_realtime_ticket_inventory`
* `query_realtime_schedule_status`
* `query_realtime_price`

---

## 5.3 门店实时能力

适用于 `store` 实体。

建议能力包括：

* `query_store_open_status`
* `query_store_capacity_status`
* `query_store_queue_status`

---

## 5.4 预约实时能力

适用于需要预约的场景。

建议能力包括：

* `query_reservation_availability`
* `query_time_slot_availability`
* `query_booking_status`

---

## 6. 实体粒度设计

实时层不应该只按“工具名”组织，还要按**查询粒度**设计。

## 6.1 推荐支持的粒度

### 实体级

例如：

* 某门店当前是否营业
* 某活动当前是否可订

输入一般是：

* `entity_type`
* `entity_id`

---

### 子资源级

例如：

* 某场次是否还有票
* 某 SKU 是否有库存
* 某时间窗是否还能预约

输入一般是：

* `entity_id`
* `sub_entity_id`
* `date/time_slot`

---

### 候选集合级

例如：

* 对 Top10 活动批量确认当前余票
* 对 Top5 门店批量确认当前营业状态

输入一般是：

* `entity_ids`

这也是最推荐的 Agent 使用方式。

---

## 7. 调用时机设计

这是实时层设计中最关键的部分之一。

## 7.1 什么时候应该调用实时层

建议在以下场景调用。

### 7.1.1 用户显式要求当前状态

典型信号：

* 现在
* 今天
* 明天
* 当前
* 还能订吗
* 还有票吗
* 还营业吗
* 还能预约吗
* 当前价格是多少

---

### 7.1.2 最终答案必须依赖当前真相

例如：

* “帮我推荐今天还能去的地方”
* “帮我看几个现在还开门的门店”
* “明天还能订的活动有哪些”

---

### 7.1.3 搜索结果需要二次确认

例如：

* 结构化层找出一批候选
* 其中只有部分对象需要确认是否当前可用

---

## 7.2 什么时候不应该调用实时层

### 7.2.1 泛浏览式推荐

例如：

* “周末带孩子去哪玩”
* “推荐几个性价比高的团购”
* “上海有哪些值得逛的商场”

这类场景通常先不查实时。

---

### 7.2.2 规则说明型问答

例如：

* “退款规则是什么”
* “适合几岁孩子”
* “停车收费规则是什么”

这类场景应优先知识层。

---

### 7.2.3 无候选对象的大范围扫描

例如：

* 直接对全上海所有活动查实时余票
* 直接对所有门店查当前状态再排序

一般不建议这样做。
成本高，时延高，也不利于稳定性。

---

## 7.3 推荐触发策略

建议按以下策略：

```text id="rt_trigger_01"
先判断是否需要实时
-> 若需要，先看是否已有候选
-> 若无候选，先做结构化搜索
-> 对 TopN 候选做实时确认
-> 返回融合结果
```

---

## 8. 接口设计

实时层最好对 Agent 提供统一风格的接口，而不是让 Agent 直接拼接底层 RPC。

## 8.1 统一接口设计原则

每个实时接口建议至少满足：

1. 输入 schema 清晰
2. 支持批量查询
3. 返回时间戳
4. 返回状态码和部分失败信息
5. 支持 trace id
6. 支持超时和降级标识

---

## 8.2 通用请求模型

建议统一抽象一个通用请求模型：

```json id="rt_req_common_01"
{
  "entity_type": "event",
  "entity_ids": ["evt_1001", "evt_1002"],
  "sub_entity_ids": [],
  "query_type": "availability",
  "time_context": {
    "date": "2026-03-21",
    "datetime": "2026-03-20T10:00:00+08:00"
  },
  "trace_id": "req_xxx",
  "timeout_ms": 1200
}
```

---

## 8.3 通用响应模型

```json id="rt_resp_common_01"
{
  "items": [
    {
      "entity_id": "evt_1001",
      "status": "available",
      "price": 19900,
      "business_status": "",
      "inventory_count": 12,
      "query_ts": "2026-03-20T10:00:01+08:00",
      "success": true
    }
  ],
  "partial_failed_ids": [],
  "query_meta": {
    "source": "realtime_availability_service",
    "latency_ms": 183,
    "degraded": false
  }
}
```

---

## 9. 具体接口类型设计

## 9.1 `query_realtime_inventory`

### 适用场景

* 商品库存
* SKU 库存
* 套餐剩余量

### 请求示例

```json id="rt_inv_req_01"
{
  "entity_type": "product",
  "entity_ids": ["prod_1001", "prod_1002"],
  "query_type": "inventory",
  "trace_id": "req_001",
  "timeout_ms": 800
}
```

### 响应示例

```json id="rt_inv_resp_01"
{
  "items": [
    {
      "entity_id": "prod_1001",
      "inventory_status": "available",
      "inventory_count": 8,
      "sell_status": "on_sale",
      "query_ts": "2026-03-20T10:00:01+08:00",
      "success": true
    }
  ]
}
```

---

## 9.2 `query_realtime_price`

### 适用场景

* 当前价格
* 当前优惠价
* 当前票档价格

### 响应应至少包含

* `entity_id`
* `price`
* `currency`
* `price_label`
* `query_ts`

---

## 9.3 `query_store_open_status`

### 适用场景

* 当前是否营业
* 是否暂停营业
* 是否停止接待

### 响应应至少包含

* `entity_id`
* `business_status`
* `open_now`
* `query_ts`

建议 `business_status` 规范化为枚举，例如：

* `open`
* `closed`
* `paused`
* `full`
* `unknown`

---

## 9.4 `query_realtime_availability`

### 适用场景

* 活动是否还能订
* 某日期是否还有余票
* 当前是否可预约

### 请求建议支持

* `entity_ids`
* `date`
* `time_slot`
* `ticket_type_id`

### 响应建议至少包含

* `entity_id`
* `availability_status`
* `available_count`
* `bookable`
* `query_ts`

---

## 10. 状态字段规范设计

为了便于 Response Composer 融合，建议实时层结果尽量统一字段语义。

## 10.1 库存状态建议枚举

* `available`
* `low_stock`
* `sold_out`
* `off_shelf`
* `unknown`

---

## 10.2 预约状态建议枚举

* `bookable`
* `almost_full`
* `full`
* `not_opened`
* `closed`
* `unknown`

---

## 10.3 营业状态建议枚举

* `open`
* `closed`
* `paused`
* `full`
* `stop_entry`
* `unknown`

---

## 10.4 价格状态建议

尽量避免只返回一个数字。
建议返回：

* `price`
* `original_price`
* `discount_price`
* `price_label`
* `price_valid_from`
* `price_valid_to`

这样后面更容易解释“当前价”和“常规价”的关系。

---

## 11. 与 Tool Router 的协作方式

## 11.1 Router 负责是否触发

Tool Router 判断：

* 是否需要实时
* 对哪些实体查实时
* 查多少个候选
* 哪些实时查询是可选步骤

---

## 11.2 实时执行节点负责实际查询

例如：

* `query_realtime`
* 或拆分成 `query_inventory` / `query_store_status`

这些节点读取 `tool_plan` 执行，不自己重新做高层路由。

---

## 11.3 推荐只对 TopN 调实时

建议由策略控制：

```text id="rt_router_01"
max_realtime_candidates = 5 / 10 / 20
```

超过上限时：

* 截断到 TopN
* 或优先对更高分候选查实时
* 或只查用户关注对象

---

## 12. 限流、超时与缓存设计

这是实时层能否生产落地的关键。

## 12.1 限流原则

实时接口通常成本高，因此建议：

1. 单请求最大实体数限制
2. 单用户短时间频控
3. 全局服务限流
4. 高峰期优先核心查询

---

## 12.2 超时原则

实时层不应无限等待。
建议每类接口设定明确预算。

### 建议预算示例

* 门店营业状态：300ms ~ 800ms
* 实时库存：500ms ~ 1000ms
* 场次可预约状态：800ms ~ 1500ms
* 复杂动态定价：800ms ~ 1500ms

具体预算要按业务系统能力来定，但 Agent 层必须有超时边界。

---

## 12.3 缓存策略

实时层不是完全不能缓存，而是要区分缓存类型。

### 可短缓存的场景

* 门店当前营业状态
* 某些低频价格
* 当前是否满位

### 不适合长缓存的场景

* 高频库存
* 高频票务余量
* 秒级价格变动

---

## 12.4 建议缓存层次

### L1：请求级去重缓存

同一轮请求内相同实体不要重复查。

### L2：短时共享缓存

例如 10 秒、30 秒、60 秒级别，按业务允许范围设置。

### L3：失败回退缓存

在实时服务暂时失败时，可用最近一次成功结果兜底，但必须带上“可能延迟”标识。

---

## 13. 降级与异常处理设计

## 13.1 降级目标

降级的目标不是“假装成功”，而是：

**在保持事实边界清晰的前提下，尽量给用户一个仍然可用的回答。**

---

## 13.2 常见异常类型

建议统一分为：

1. 超时
2. 部分失败
3. 全量失败
4. 空返回
5. 数据异常
6. 限流拒绝

---

## 13.3 超时降级

### 动作建议

* 使用结构化快照结果继续回答
* 添加 disclaimer
* 记录 trace
* 标记 `realtime_status=timeout`

### 示例提示

* “实时状态查询超时，以下结果基于快照信息，实际可用情况可能有变化。”

---

## 13.4 部分失败降级

例如查 10 个候选，只成功 7 个。

### 动作建议

* 成功结果正常融合
* 失败对象保留快照信息
* 明确哪些对象未完成实时确认

### 推荐状态

* `realtime_status = partial_success`

---

## 13.5 全量失败降级

### 动作建议

* 若业务允许，回退到快照结果
* 若问题本身强依赖实时，则应明确告知无法确认当前状态

### 示例

用户问：

“现在还能订吗？”

如果实时全失败，不能直接说“能订”。
只能说：

“当前没有成功查询到实时可订状态，因此我不能确认现在是否还能订。”

这点非常重要。

---

## 13.6 限流降级

### 动作建议

* 降低候选数
* 跳过可选实时步骤
* 保留核心实体的实时确认
* 其余结果回退到快照

---

## 14. 与 Response Composer 的融合原则

实时结果进入最终回答时，建议遵守以下原则。

## 14.1 实时状态优先覆盖快照状态

如果实时结果存在，则优先使用实时结果中的：

* 可售状态
* 营业状态
* 可预约状态
* 当前价格

---

## 14.2 无实时结果时不要伪造

不要把“未查到实时”说成“实时正常”。

---

## 14.3 实时时间戳应可用

最终回答或卡片层最好保留：

* `query_ts`
* 或“刚刚更新 / 当前查询时间”

这样更有助于用户理解状态时效性。

---

## 14.4 强实时问法必须更保守

像这些问题：

* “现在还有吗”
* “今天还能订吗”
* “当前营业吗”

如果没有实时结果，回答必须保守。
不能只依赖知识层或快照层给肯定判断。

---

## 15. Agent 执行链路中的实时层模式

## 15.1 搜索后实时确认

```text id="rt_flow_01"
parse_intent
-> normalize_params
-> route_tools
-> execute_structured_search
-> query_realtime
-> compose_response
```

适用场景：

* 明天还能订的活动
* 当前还营业的门店
* 当前有库存的商品

---

## 15.2 搜索后并行实时 + 知识

```text id="rt_flow_02"
parse_intent
-> normalize_params
-> route_tools
-> execute_structured_search
-> parallel(
    query_realtime,
    retrieve_knowledge
  )
-> compose_response
```

适用场景：

* 推荐几个现在还能去的活动，并说明入场规则

---

## 15.3 单实体实时确认

```text id="rt_flow_03"
parse_intent
-> normalize_params
-> route_tools
-> query_realtime
-> compose_response
```

适用前提：

* 会话上下文里已有明确实体
* 例如用户问“这个店现在开着吗”

---

## 16. 数据结构定义建议

## 16.1 RealtimeQueryRequest


## 17. 评估与监控建议

实时层必须单独治理，不然很容易拖垮整个 Agent 体验。

## 17.1 服务指标

建议监控：

* QPS
* P95 / P99 延迟
* 超时率
* 错误率
* 部分失败率
* 限流率
* 缓存命中率

---

## 17.2 业务指标

建议监控：

* 实时查询触发率
* 每次请求平均实时实体数
* 因实时失败触发的降级率
* 实时结果覆盖率
* 实时冲突率

---

## 17.3 用户体验指标

建议关注：

* 强实时问题回答成功率
* “现在 / 今天 / 明天”类问题满意度
* 因状态错误导致的投诉率
* 二次追问率

---

## 18. 最终结论

实时层在整个系统中的正确定位是：

**一个面向当前状态事实查询的高时效能力层。**

它不替代：

* 结构化层的大规模召回与排序
* 知识层的规则与说明解释
* 业务系统的底层主数据治理

它的核心价值在于：

1. 把“当前能不能”“现在是不是”“还有没有”这类问题答准
2. 给结构化结果提供最后一步状态确认
3. 在高时效场景下避免用过期快照误导用户
4. 让 Agent 从“会推荐”进化到“能确认”
