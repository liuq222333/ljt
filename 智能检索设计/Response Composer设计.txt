# 《Response Composer 详细设计 V1》

这份文档的目标，是把前面已经定义好的三类能力层真正收敛成一个**可控的最终回答生成模块**。
如果说：

* Tool Router 决定“查什么”
* 结构化层 / 知识层 / 实时层负责“给事实”
* 那么 Response Composer 负责“怎么把事实变成可信、好读、可继续交互的回答”

---

# 一、Response Composer 详细设计 V1

## 1. 文档目标

本文档定义 Agent 架构中 Response Composer 的职责、输入输出、融合策略、冲突处理、推荐理由生成、回答 schema 和下一轮引导机制。

Response Composer 的核心目标是：

**将多来源结果融合为面向用户的最终回答，同时保持事实边界清晰、冲突可解释、输出结构稳定。**

它解决的问题包括：

* 结构化结果怎么展示
* 知识结果怎么补充
* 实时结果怎么覆盖快照
* 多来源冲突时怎么判断
* 推荐理由怎么写才不“瞎夸”
* 用户下一轮最自然的引导怎么给

它不解决的问题包括：

* 原始意图理解
* 参数标准化
* 工具路由
* 底层检索 / 查询执行

---

## 2. 设计定位

在整体链路中，Response Composer 位于最后一段：

```text id="rc_arch_01"
parse_intent
-> normalize_params
-> route_tools
-> execute tools
-> compose_response
-> final answer
```

它是：

**结果融合器 + 回答结构生成器 + 风险提示生成器**

而不是一个“自由发挥的 LLM 总结器”。

---

## 3. 职责边界

## 3.1 Response Composer 负责什么

### 3.1.1 多来源结果融合

输入可能同时来自：

* 结构化层
* 知识层
* 实时层
* session 上下文
* tool 执行轨迹

Response Composer 需要把这些结果组织成统一输出。

---

### 3.1.2 证据优先级控制

不同来源之间可能冲突。
Response Composer 需要按既定优先级做覆盖和解释。

---

### 3.1.3 推荐理由生成

对推荐型场景，需要生成基于证据的推荐理由，例如：

* 符合用户条件
* 当前可订
* 评分较高
* 距离较近
* 适合亲子
* 支持退款

---

### 3.1.4 风险提示与 disclaimer 生成

例如：

* 实时查询超时
* 结果基于快照
* 规则文档未命中
* 某些候选未完成实时确认

---

### 3.1.5 输出结构标准化

无论底层调用了多少工具，最终都尽量输出统一 schema，便于：

* 前端渲染
* 卡片展示
* 多端复用
* 埋点分析
* A/B 测试

---

### 3.1.6 下一轮引导生成

例如：

* 是否继续按价格筛选
* 是否继续按距离筛选
* 是否查看退款规则
* 是否确认具体门店 / 活动

---

## 3.2 Response Composer 不负责什么

### 3.2.1 不负责决定调什么工具

这是 Tool Router 的职责。

### 3.2.2 不负责底层事实真相生成

它不能“发明事实”。
只能融合现有证据。

### 3.2.3 不负责业务规则本身

例如退款规则是什么，不是它定义的。
它只能引用知识层结果。

---

## 4. 设计原则

## 4.1 事实优先于表达

回答可以流畅，但不能为了好听牺牲事实边界。

---

## 4.2 来源优先级固定

建议统一采用：

```text id="rc_priority_01"
实时层 > 结构化层快照 > 知识层文档 > LLM 推断
```

LLM 只能做：

* 组织语言
* 做归纳
* 做解释性连接

不能把推断提升为事实。

---

## 4.3 推荐理由必须可追溯

推荐理由必须尽量能落到某个事实来源，例如：

* 来自结构化层评分字段
* 来自实时层营业状态
* 来自知识层适合人群说明

不要生成纯主观理由，例如：

* “我觉得这个更适合你”
* “这个看起来很不错”

---

## 4.4 冲突不隐藏

多来源冲突时，不应静默覆盖且不解释。
至少在高风险信息上，应明确说明。

---

## 4.5 强实时问题更保守

对于“现在 / 今天 / 明天”类问题，如果没有可靠实时结果，输出要保守。

---

## 4.6 输出尽量结构化

先产出结构化 answer，再渲染成文本。
不要反过来只生成一大段文本再硬拆结构。

---

## 5. 输入设计

## 5.1 输入来源

Response Composer 的输入建议包括：

```json id="rc_input_01"
{
  "parsed_intent": {},
  "normalized_params": {},
  "search_results": {},
  "retrieved_docs": {},
  "realtime_results": {},
  "tool_plan": {},
  "session_context": {},
  "execution_meta": {},
  "debug_trace": []
}
```

---

## 5.2 核心输入字段说明

### `parsed_intent`

用于判断：

* 回答类型
* 是否要推荐
* 是否要解释
* 是否是强实时问题

---

### `normalized_params`

用于判断：

* 用户显式条件有哪些
* 哪些条件是推断来的
* 默认条件有哪些
* 推荐理由里哪些能说

---

### `search_results`

提供：

* 候选对象
* 排名
* 快照字段
* 基础展示信息

---

### `retrieved_docs`

提供：

* 规则说明
* 注意事项
* 适合人群
* 停车 / 交通 / 入园等说明
* FAQ 支撑内容

---

### `realtime_results`

提供：

* 当前价格
* 当前库存
* 当前营业状态
* 当前可预约状态
* 当前余票状态

---

### `tool_plan` 与 `execution_meta`

用于识别：

* 哪些步骤是可选的
* 哪些步骤已降级
* 哪些结果可能不完整

---

## 6. 输出目标

Response Composer 的输出，建议分成三层：

### 第一层：结构化 answer object

供前端与系统消费。

### 第二层：自然语言 answer_text

供直接展示给用户。

### 第三层：补充 UI 元素

例如：

* 卡片
* badges
* disclaimers
* next actions

---

## 7. 输出 schema 设计

## 7.1 推荐总 schema

```json id="rc_output_01"
{
  "answer_type": "recommendation",
  "answer_text": "",
  "summary": "",
  "cards": [],
  "disclaimers": [],
  "citations": [],
  "next_actions": [],
  "composer_meta": {
    "used_sources": [],
    "conflict_detected": false,
    "degraded": false
  }
}
```

---

## 7.2 字段说明

### `answer_type`

建议枚举：

* `recommendation`
* `search_result`
* `faq_answer`
* `rule_explanation`
* `realtime_confirmation`
* `clarification`
* `no_result`
* `partial_result`

---

### `answer_text`

最终面向用户的主回答文本。

---

### `summary`

用于更短的摘要层，可给前端做标题、副标题或首句。

---

### `cards`

候选对象卡片数组，是推荐和搜索场景最重要的结构化输出。

---

### `disclaimers`

风险提示、时效提示、降级提示。

---

### `citations`

引用来源信息，便于透明性与可核验性。

---

### `next_actions`

下一轮建议交互入口。

---

### `composer_meta`

内部元信息，用于埋点、调试和治理。

---

## 8.1 推荐 card schema

```json id="rc_card_01"
  "entity_type": "event",
  "title": "某某亲子展",

  "status_info": {},
  "score_info": {},

```

---

## 8.2 字段说明

### `highlights`

展示事实型亮点。
例如：

* “室内场馆”
* “评分较高”
* “亲子标签匹配”

---

### `reason_tags`

用于显示推荐理由标签。
例如：

* “亲子友好”
* “当前可订”
* “退款规则清晰”

---

### `realtime_badges`

用于显示实时状态。
例如：

* “当前可订”
* “正在营业”
* “余票紧张”
* “库存不足”

---

### `knowledge_notes`

用于显示知识层补充说明。
例如：

* “入园需成人陪同”
* “活动开始前24小时可退”
* “建议提前预约”

---

### `price_info`

建议结构：

```json id="rc_price_info_01"

  "current_price": 19900,
  "original_price": 25900,
  "currency": "CNY",
  "price_label": "当前价"

```

---

### `status_info`

建议结构：

```json id="rc_status_info_01"
{

  "availability_status": "bookable",

  "query_ts": "2026-03-20T10:00:01+08:00"
}
```

## 9. 回答类型设计

Response Composer 不应对所有场景用同一种模板。
建议至少分下面几类。

## 9.1 推荐型回答

适用于：

* 商品推荐
* 活动推荐
* 门店推荐

特点：

* 重点输出 TopN 卡片
* 给出推荐理由
* 提供下一轮筛选建议

---

## 9.2 FAQ / 规则型回答

适用于：

* 退款规则
* 停车规则
* 入园须知
* 是否适合某类人群

特点：

* 直接回答
* 用知识证据支撑
* 必要时补充限制条件

---

## 9.3 实时确认型回答

适用于：

* 现在还能订吗
* 当前营业吗
* 还有库存吗

特点：

* 优先输出明确状态
* 强调实时查询时间
* 没查到就保守表达

---

## 9.4 搜索结果型回答

适用于：

* 用户是浏览 / 查找而非强推荐
* 更强调“查到哪些”

---

## 9.5 无结果型回答

适用于：

* 结构化层空结果
* 实时筛完后无可用对象

特点：

* 明确没有结果
* 不伪造候选
* 提供条件调整建议

---

## 9.6 部分结果型回答

适用于：

* 部分实时失败
* 部分知识缺失
* 结果不完整但仍可回答

特点：

* 有结果
* 但要带 disclaimer

---

## 10. 多来源融合策略

这是 Response Composer 的核心。

## 10.1 融合总体流程

建议流程如下：

```text id="rc_fusion_01"
收集候选对象
-> 建立 entity merge map
-> 注入实时结果
-> 注入知识结果
-> 处理冲突
-> 生成推荐理由
-> 生成文本与卡片
-> 生成 disclaimers 和 next actions
```

---

## 10.2 Entity Merge Map

建议以 `entity_id + entity_type` 为主键，把不同来源结果合并到同一对象上。

例如：

```json id="rc_merge_01"
{
  "entity_key": "event:evt_1001",
  "snapshot": {},
  "realtime": {},
  "knowledge": []
}
```

这样后面更容易做：

* 实时覆盖
* 知识补充
* card 生成

---

## 10.3 快照结果注入

结构化层通常提供：

* title
* subtitle
* score
* distance
* rating
* tags
* 类目
* 快照状态

这部分一般构成 card 的主体骨架。

---

## 10.4 实时结果注入

实时层结果优先覆盖以下字段：

* 当前价格
* 当前可订状态
* 当前营业状态
* 当前库存状态
* 当前预约状态

若实时结果缺失，则保留快照结果，但要标记“未实时确认”。

---

## 10.5 知识结果注入

知识层结果主要注入：

* 规则说明
* 注意事项
* 适合人群
* 退款说明
* 入园说明
* 停车说明

知识结果不应直接覆盖结构化主字段，通常作为补充说明存在。

---

## 11. 冲突处理设计

## 11.1 冲突类型

建议至少识别以下冲突：

### A. 实时 vs 快照冲突

例如：

* 快照显示可售，实时显示售罄
* 快照显示营业，实时显示暂停营业

---

### B. 实时 vs 知识冲突

例如：

* 知识写“通常营业到 22:00”
* 实时显示当前暂停营业

---

### C. 结构化 vs 知识冲突

例如：

* 结构化标签写“适合儿童”
* 知识说明却写“更适合 8 岁以上”

---

### D. 多文档知识冲突

例如：

* 两个版本规则同时命中
* 一个通用规则，一个实体专属规则

---

## 11.2 冲突优先级规则

建议固定如下：

```text id="rc_conflict_01"
1. 实时层优先于结构化层
2. 结构化层优先于知识层中的泛描述字段
3. 实体专属知识优先于通用知识
4. 当前生效版本优先于旧版本
5. 高优先级文档优先于低优先级文档
```

---

## 11.3 冲突处理动作

### 强事实冲突

例如库存、营业状态、可订状态。
处理方式：

* 直接以高优先级来源覆盖
* 必要时生成说明

### 软解释冲突

例如适合人群、体验说明。
处理方式：

* 尽量选择更具体、更近的实体知识
* 避免把冲突内容同时强行写进推荐理由

---

## 11.4 冲突说明示例

例如：

“常规营业说明显示该门店通常营业到晚上 10 点，但我刚查到的当前状态是暂停营业，所以请以当前实时状态为准。”

---

## 12. 推荐理由生成设计

这是用户感知最强的部分之一。

## 12.1 推荐理由来源

推荐理由应来自以下事实：

### 结构化层

* 排名靠前
* 评分高
* 热度高
* 距离近
* 类目匹配
* 标签匹配

### 实时层

* 当前可订
* 当前营业
* 当前有库存
* 当前价格更优

### 知识层

* 适合亲子
* 支持退款
* 入场门槛低
* 雨天也适合
* 低龄友好

---

## 12.2 推荐理由生成原则

### 原则 1：优先使用用户显式条件

如果用户说“适合低龄儿童”，那“适合低龄儿童”可以进理由。
如果用户没说，不要硬凑。

### 原则 2：优先事实，不要抽象赞美

少说：

* “非常棒”
* “超级值得”
* “我觉得你会喜欢”

多说：

* “室内场馆，适合雨天”
* “当前可订”
* “支持退款”
* “评分较高”

### 原则 3：每张 card 的理由不宜过多

建议每个卡片保留 2 到 4 个最强理由。

---

## 12.3 推荐理由模板建议

### 推荐型场景

可组合成：

* 符合你的 `{显式条件}`
* `{结构化特征}`
* `{实时状态}`
* `{知识补充}`

例如：

* “符合你要找的亲子活动，当前还能订，且支持活动前 24 小时退款。”

---

## 13. Disclaimer 设计

disclaimer 很重要，因为系统并不总能拿到完整信息。

## 13.1 建议触发 disclaimer 的场景

### 实时超时

例如：

* 实时接口超时
* 结果回退到快照

---

### 部分失败

例如：

* 只查到一部分候选的实时状态

---

### 知识缺失

例如：

* 没查到规则说明
* 无法确认某类限制条件

---

### 强时效问题但结果不完整

例如：

* 用户问“现在还能订吗”
* 但实时全失败

---

## 13.2 disclaimer 示例

* “实时状态查询超时，以下可用性主要基于快照信息，实际情况可能有变化。”
* “其中部分结果未完成实时确认，我已经优先展示了查到当前状态的候选。”
* “没有检索到明确的退款规则，以下仅基于搜索结果和已知说明整理。”

---

## 14. Citation 设计

如果系统需要透明性和可核验性，建议 Response Composer 输出 citations。

## 14.1 citation 来源类型

* `structured_snapshot`
* `knowledge_doc`
* `realtime_service`

---

## 14.2 推荐 citation 结构

```json id="rc_citation_01"
{
  "source_type": "knowledge_doc",
  "source_id": "doc_1001",
  "title": "退款规则",
  "entity_id": "evt_1001",
  "snippet": "活动开始前24小时可退"
}
```

---

## 14.3 使用原则

并不是每句都要引用。
但这些内容建议可追踪：

* 规则型结论
* 强实时结论
* 高风险限制条件
* 关键推荐理由

---

## 15. No Result 与 Partial Result 设计

## 15.1 No Result 场景

### 结构化层无结果

例如：

* 条件太窄
* 城市 / 时间 / 类目组合无命中

### 实时筛完无结果

例如：

* 原本有候选
* 但“今天还能订”的实时确认后全部不可用

---

## 15.2 No Result 输出原则

* 明确告诉用户当前没有符合条件的结果
* 说明是哪个条件限制住了结果
* 给出 1 到 2 个自然的放宽建议

### 示例

“我这次没有筛到符合‘明天还能订 + 亲子 + 上海’这几个条件的活动。你可以试试放宽到本周末，或者先不限制必须明天可订。”

---

## 15.3 Partial Result 场景

例如：

* 找到了候选
* 但实时只确认了一部分
* 或规则说明只覆盖了一部分对象

输出应包含：

* 可用结果
* 不完整提示
* 下一步建议

---

## 16. 下一轮引导设计

这个模块会直接影响多轮体验。

## 16.1 引导原则

### 原则 1：紧贴当前任务

不要给太散的引导。
要围绕当前结果继续收敛。

### 原则 2：优先提供“最自然的下一步”

例如：

* 按价格缩小
* 按距离缩小
* 看退款规则
* 看当前营业状态
* 只看室内

### 原则 3：数量少一点

建议 2 到 4 个 next actions 即可。

---

## 19. 工程实现建议

## 19.1 建议分两步生成

不要一上来让 LLM 直接把所有东西写成最终回答。
建议：

### 第一步：规则化融合

先用程序逻辑完成：

* entity merge
* 字段覆盖
* 冲突识别
* 推荐理由候选生成
* disclaimer 触发判断

### 第二步：LLM 负责表达层

再把融合后的结构化结果交给 LLM 生成：

* `answer_text`
* `summary`
* 更自然的 next action 文案

这样更稳。

---

## 19.2 建议保留“pre_answer_object”

在真正生成 `final_answer` 之前，建议先有一层中间对象：

```json id="rc_pre_answer_01"
{
  "merged_entities": [],
  "conflicts": [],
  "reason_candidates": [],
  "disclaimer_candidates": [],
  "answer_type": "recommendation"
}
```

这样更便于：

* debug
* A/B 实验
* 规则升级
* 后续训练数据沉淀

---

## 19.3 强约束尽量程序化

这些不要完全交给 LLM：

* 实时优先级覆盖
* conflict resolution
* no-result 判定
* disclaimer 触发
* citation 选择

这些应该程序化。

---

## 20. 评估与监控建议

Response Composer 单独也需要评估。

## 20.1 质量指标

建议评估：

* 推荐理由真实性
* 冲突处理正确率
* disclaimer 触发准确率
* final answer 结构完整率
* next action 点击率
* 无依据夸大率

---

## 20.2 用户体验指标

建议关注：

* 首答满意度
* 二次追问率
* 卡片点击率
* 下一轮建议采纳率
* 因错误表述导致的投诉率

---

## 20.3 治理指标

建议增加：

* 幻觉型推荐理由比例
* 未引用高风险事实比例
* 强实时问题误判率
* 冲突静默覆盖率

---

## 21. 最终结论

Response Composer 在整个系统中的正确定位是：

**一个基于证据优先级和结构化融合规则的最终回答生成层。**

它不是简单的“最后让 LLM 总结一下”，而是：

1. 统一融合结构化、知识和实时结果
2. 处理多来源冲突
3. 生成可追溯的推荐理由
4. 产出稳定的最终回答 schema
5. 给出自然的下一轮交互入口

它决定了用户最终感知到的：

* 系统是否可信
* 推荐是否有依据
* 冲突是否解释清楚
* 多轮是否顺滑
