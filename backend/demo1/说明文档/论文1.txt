基于事件驱动与大模型 Agent 的社区服务智能触达系统设计与实现
摘要
面向社区服务场景中高频通知、协同服务与内容分发需求，传统触达机制通常依赖固定规则与人工配置，难以在多业务、多条件环境下实现精准与灵活并存的触达效果。为解决上述问题，本文围绕“事件驱动 + 大模型 Agent + 受控工具调用”的总体思路，设计并实现一套社区服务智能触达系统。系统以业务事件与用户交互为输入入口，通过大模型对意图与上下文进行理解与策略决策，再由后端工具执行层依据接口白名单与路由配置执行真实业务接口，实现从“语义理解”到“可控执行”的闭环。项目后端采用 Spring Boot 构建基础服务框架，使用 MyBatis/MyBatis-Plus 完成数据持久化，结合 RabbitMQ 实现通知消息的异步投递与削峰，使用 Redis 实现未读数缓存、短信验证码限流与地理检索加速，并通过 WebSocket/STOMP 实现未读通知实时推送；同时引入 MinIO 作为对象存储，完成社区动态与商品图片、用户头像的上传与预签名访问。项目还实现了社区动态、社区集市、本地活动、邻里互助等业务模块，并提供管理端接口用于路由白名单治理、用户管理、内容审核与系统通知发布。本文对系统架构、核心模块、关键流程与实现细节进行详细阐述，并给出可复现的功能验证与评估方法，为社区类应用在大模型时代的智能触达与可控落地提供参考。
关键词：事件驱动；大模型 Agent；工具调用；智能触达；社区服务；RabbitMQ；Redis；WebSocket；MinIO
一 引言
（1）研究背景
随着社区数字化服务体系不断完善，商品发布、邻里互助、社区活动、举报与审核、系统公告等业务日益频繁。社区场景的触达具有明显特点：触达对象多样（普通用户、卖家、志愿者、管理员）、触达时机多变（事件发生即触达、活动开始前提醒、审核结果通知）、触达渠道多样（站内通知、实时推送、公告列表），且触达策略需要考虑上下文（用户身份、地理位置、兴趣偏好、频控与优先级）。传统触达系统通常依赖固定规则，如关键词匹配、简单标签过滤或硬编码分支逻辑；当业务扩展时，规则数量膨胀、冲突难以管理，且难以表达复杂的“意图到动作”的映射关系。
另一方面，大模型具备自然语言理解与策略推理能力，可以从用户输入或业务事件中抽取意图、补全参数并选择下一步动作。但若允许模型直接调用业务接口，容易引入越权访问、错误参数、不可解释行为与审计困难等风险。因此，社区服务场景需要一种既具备模型智能决策能力，又具备后端强约束与可治理机制的触达系统。
（2）研究问题
本文主要解决以下问题：1. 如何将社区业务中的“事件触发”与“触达执行”解耦，实现异步化与可扩展；2. 如何利用大模型 Agent 对自然语言与业务上下文进行理解，输出可执行的策略与动作；3. 如何以接口白名单与路由治理限制模型的可调用范围，确保安全、可控与可审计；4. 如何在实际项目中落地通知链路、对象存储、地理检索与管理端功能，形成完整可运行的工程实现。
（3）研究内容与贡献
本文在一个可运行的社区后端项目中实现并验证以下内容：1. 设计并实现基于 api_routes 的接口白名单与路由治理机制，为工具调用提供“受控执行”边界；2. 实现基于 DeepSeek Chat Completions 的 Agent 对话入口与工具调用闭环，通过统一 backend_api 工具完成路由查表、参数拼装与接口转发；3. 构建基于 RabbitMQ 的通知消息投递链路，并结合 Redis 缓存与 WebSocket 推送实现“落库 + 未读数实时推送”的端到端闭环；4. 完成社区动态、社区集市、本地活动、邻里互助、登录注册与用户资料等业务模块实现，并提供管理端审核与治理接口，覆盖项目主要功能面。
二 相关技术与理论基础
（1）事件驱动架构（EDA）
事件驱动架构通过“事件生产—事件消费”解耦系统模块，使系统能够以异步方式处理业务高峰并增强扩展性。在本项目中，通知系统采用 RabbitMQ 将“产生通知”与“通知落库/推送”解耦：业务侧将 NotificationMessage 投递到交换机，消费侧监听队列并完成持久化与推送，从而避免业务主链路被通知写库与推送阻塞。与此同时，消息队列还支持重试、死信等机制，为稳定性与可观测性提供基础[3][5][8]。
（2）大模型 Agent 与工具调用机制
大模型 Agent 通过 Function Calling/Tool Calling 机制将自然语言推理结果转化为结构化调用参数，从而与外部系统交互。本文采用“模型选择动作，后端执行动作”的分层思想：模型只产生 backend_api 的 resource/action 与参数建议，后端基于路由白名单校验与参数验证后再执行真实接口调用，实现模型能力与系统安全之间的平衡[15][16][17]。
（3）Spring Boot、MyBatis/MyBatis-Plus 与工程化实现
项目基于 Spring Boot 构建 Web 服务与组件管理，使用 MyBatis-Plus 以及 MyBatis 注解/XML 映射实现对 MySQL 的数据访问，并使用 springdoc-openapi（@Operation）为接口提供可读的文档描述。对于需要分页与筛选的场景，项目引入 PageHelper 进行分页支持。安全层面使用 Spring Security 提供密码加密（BCryptPasswordEncoder）与基础 CORS 配置[7][11][12]。
（4）Redis、RabbitMQ、WebSocket 与对象存储
Redis 在本项目中承担多种角色：用于通知未读数缓存、短信验证码存储与限流锁、商品与活动的地理检索与 RediSearch 索引。RabbitMQ 用于通知消息异步投递与分发，并配套死信队列实现异常隔离。WebSocket/STOMP 用于向前端推送未读通知增量，提升交互实时性。MinIO 作为对象存储用于保存动态图片、商品图片与用户头像，并通过预签名 URL 控制访问有效期，避免直接暴露存储密钥[4][5][6][9][10][13][14]。
三 需求分析与系统建模
（1）业务需求分析
社区系统在业务层需要覆盖内容发布、交易协作、活动组织与互助协同等典型场景。项目实现的核心业务包含：社区动态发布与互动（发布动态、点赞、评论、删除）、社区集市商品发布与交易（商品发布、库存与价格调整、购物车、下单与订单查询）、本地活动发布与管理（活动创建进入审核、审核通过后发布、用户报名记录查询、活动附近检索与热力图展示、邻里故事发布与阅读）、邻里互助任务（发布互助任务进入审核、审核通过后公开、任务列表查询）、通知与公告（系统/业务通知落库、已读管理、未读数统计、实时推送）、用户与登录（注册、登录、短信验证码、修改密码、用户资料更新与头像管理）以及管理端治理能力（接口白名单管理、用户管理、活动/互助审核、管理员删除商品、管理员发布系统通知）。
（2）功能需求分析
从系统功能角度，本项目需要实现“触达闭环”与“业务闭环”两类能力。触达闭环包含：通知消息产生、异步投递、消费落库、未读数维护、实时推送与读取/已读/删除等全链路能力。业务闭环包含：商品、动态、活动与互助任务等实体的创建、查询、更新与审核流转，同时要求业务动作能够被统一纳入工具调用体系，便于 Agent 在统一入口中按需触发。
（3）非功能需求
系统需满足安全性、可扩展性与可维护性要求。安全性体现在接口调用必须受控，尤其是模型驱动的工具调用必须被白名单限制；同时密码需加密存储，短信验证码需限流防刷。可扩展性体现在新增业务接口可以通过 api_routes 配置快速加入工具调用范围；通知链路通过消息队列可扩展到更多队列与消费者。可维护性体现在模块职责清晰、统一响应结构 Resp、接口文档可读、并通过日志与审计信息支持定位问题[1][2]。

五 关键技术设计
（1）受控工具调用与 api_routes 白名单
系统通过 api_routes 表对“可被模型调用的接口”进行集中治理。每条路由记录包含 resource、action、http_method、path_template、path_params、operation_type、description 与 enabled 等字段。AgentToolHandler 在执行 backend_api 时必须先通过 ApiRouteService 查找 enabled=1 的路由记录，未命中则拒绝执行，从而避免模型随意访问系统内部接口。管理端 ApiManagementController 提供路由新增、更新、启用/禁用与删除能力，实现运维层面的快速治理。为了便于绘制论文中的设计图，可以将 backend_api 的参数结构（resource/action/pathVariables/params/payload/authorization）整理为结构示意图或表格，并在图中说明参数来源与校验规则[1][16]。

（2）Agent 编排与工具注入设计
AgentChatServiceImpl 构建大模型请求时，将 system prompt、历史消息与工具定义一并注入。工具定义由 AgentToolHandler 生成并以 schema 约束输入输出结构，降低模型填参歧义。为了增强可控性，系统在收到工具调用后执行参数反序列化与字段校验，并将执行结果包装为包含 data 与 presentation_hint 的结构，辅助模型组织更可读的自然语言回复。可在论文中绘制“模型请求结构 + 工具注入位置”的示意图，展示 messages 与 tools 的组合方式，以及 tool_calls 触发后的执行分支[15][16][17][18]。

（3）通知模块 RabbitMQ 拓扑设计
通知模块采用 TopicExchange 作为主交换机，支持按 routingKey 将消息分发到用户队列、群组队列与广播队列；同时配套 DirectExchange 作为死信交换机并绑定死信队列，用于消费异常时的隔离。NotificationSender 根据 NotificationMessage 的 targetType 与目标标识生成 routingKey（如 notif.user.{uid}、notif.group.{groupKey}、notif.broadcast），并支持 TTL 设置；NotificationMessageListener 监听多个队列消费消息并落库。可在论文中绘制 RabbitMQ 拓扑图，标注交换机、队列、绑定关系、routingKey 规则以及死信流转路径[3][5][8]。

（4）通知消费幂等与未读实时推送设计
通知消费侧在落库前进行幂等校验：当消息体带有 msgId 时，先查询 notifications 表中是否已存在同一 msgId 与 userId 的记录，若存在则直接跳过，避免重复落库导致的未读数错误。落库后系统对 Redis 未读 Key 进行递增，并通过 WebSocket 向对应用户的 /user/{userId}/queue/notify 推送增量。该设计使通知既可在通知列表中回溯查询，又能在前端实时感知未读变化。可绘制“消费处理流程图”展示幂等判断、落库、缓存更新与推送的顺序关系[1][4][6][9][10]。

（6）附近检索多级回退策略

七 安全与治理机制
（1）接口白名单与最小可调用面
系统通过 api_routes 将工具调用能力收敛到可配置白名单，backend_api 的每次执行都必须命中 enabled 路由记录，未命中则拒绝执行并返回错误信息。该机制使模型的“能力边界”从代码层抽离到配置层，便于审计与动态治理[1][16]。
（2）参数校验、幂等与数据一致性
工具调用层对 resource/action/路径变量进行显式校验，避免出现路径拼装错误或不完整参数导致的异常调用。通知消费侧通过 msgId+userId 查重实现幂等，避免重复投递导致的重复落库与未读数错误。未读数缓存采用 TTL 与回源机制结合：缓存命中直接返回以提升性能，缓存缺失回源数据库并重建缓存，读写操作后主动清理或重置缓存以降低一致性风险[1][4][9]。
（3）消息可靠性与异常隔离
RabbitMQ 拓扑中引入死信交换机与死信队列，用于在消息处理异常时进行隔离，避免阻塞主消费队列。消息体支持 TTL，使过期通知不会长期堆积。项目中还预留了 retry 队列，便于后续引入延迟重试策略[3][5][8]。
（4）基础安全配置与异常处理
系统通过 Spring Security 提供 BCryptPasswordEncoder 统一密码加密，并配置 CORS 允许本地前端域名访问；同时关闭 CSRF 以适配前后端分离的接口调用方式。全局异常处理器对邮箱冲突等业务异常返回结构化响应（包含时间戳、状态码与路径），提升前端可处理性[7]。
八 实验与评估
（1）功能性验证方法
针对触达链路，采用端到端验证：通过 /api/notifications/send 投递消息后，检查 notifications 表中是否落库、Redis 未读数是否递增、WebSocket 是否收到未读增量推送，并通过 /api/notifications 接口验证列表读取、/unreadCount 验证未读数读取、/markAllRead 或 /{id}/read 验证已读状态变更与缓存更新。针对商品交易链路，采用事务一致性验证：执行 /api/carts/buyCartItems 后检查库存扣减、订单生成、购物车条目删除以及买卖双方通知是否产生。针对活动与互助审核链路，验证审核表到正式表的迁移是否完整，并验证审核结果通知是否发送。

（2）性能与指标化评估方法
触达时延可定义为“消息投递时间戳到 WebSocket 推送时间戳”的差值，可通过服务端日志或消息体 createdAt 与消费时间对比测量。接口失败率可通过统计 Resp.code 非 200 的比例或通过日志聚合统计异常次数得到。附近检索性能可通过对 /api/products/nearby 与 /api/local-act/activities/nearby 的响应时间与查询命中来源（redis-geo、redisearch、sql 回退）进行对比分析。对工具调用链路，可通过模拟不同的 resource/action（正确、禁用、缺失参数）观察拒绝率与错误定位信息，验证白名单治理与参数校验有效性。
（3）评估结论（基于验证结果的总结）
通过对典型业务场景的功能验证与链路观察，本项目实现了从事件产生到触达可见的完整闭环，并在工程层面提供了可扩展的接口治理能力与多存储组件协同能力。通知链路具备异步投递、幂等落库、未读缓存与实时推送能力；业务模块具备可运行的发布、交易、审核与互动功能；工具调用机制为后续引入“事件 → Agent → 触达策略”的自动化决策提供了可控的执行底座。
九 总结与展望
（1）结论
本文基于实际项目实现了一套“事件驱动 + 大模型 Agent + 受控工具调用”的社区服务智能触达系统。系统以 api_routes 为核心提供接口白名单与治理能力，以 backend_api 工具实现路由查表与可控转发，以 RabbitMQ+Redis+WebSocket 构建通知触达闭环，并在社区动态、社区集市、本地活动、邻里互助、用户与登录、管理端审核与治理等模块上完成较为完整的业务功能实现。该方案在保持工程可维护性与安全可控的前提下，引入了大模型对意图与策略的表达能力，为社区触达系统的智能化升级提供了可落地路径。
（2）展望
未来工作可从三个方向进一步完善：1. 在工具调用链路中引入更细粒度的权限校验与策略引擎（按用户角色、资源范围、频控与风险等级控制可调用动作）；2. 将 operation_type 与候选 action 列表选择机制落地为两阶段选择流程，进一步降低模型选错接口的概率；3. 建立触达效果评估闭环，沉淀触达日志与用户反馈数据，形成可量化的策略优化机制，并逐步扩展到推荐、内容审核与协同服务等更复杂场景。
